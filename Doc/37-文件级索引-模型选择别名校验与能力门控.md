## 2. 主模型解析与优先级

## 2.1 `utils/model/model.ts`

已核对要点：

1. 主模型优先级固定：会话内 `/model` 覆盖 > 启动参数/环境 > 持久设置 > 内置默认。
2. `getMainLoopModel()` 返回最终可调用模型；`getDefaultMainLoopModelSetting()`负责订阅档位默认策略。
3. 对 Max/Team Premium 与其他订阅档位采用不同默认模型分流；3P provider 与 first-party 默认值可分离。
4. 解析中包含 alias、[1m] 上下文和 provider 命名兼容处理。

## 3. 模型可选项生成与展示

## 3.1 `utils/model/modelOptions.ts`

已核对要点：

1. 按用户类型与订阅档位构造模型候选（ant / Max / Team / Pro / PAYG）。
2. 候选项与 fast mode、1M 上下文可用性、计费提示联动。
3. 3P provider 支持基于环境变量注入自定义 Sonnet/Opus/Haiku 展示项。
4. 选项 value 与描述文案分离，便于 UI 展示和执行层解析解耦。

## 3.2 `components/ModelPicker.tsx`

已核对要点：

1. 负责模型选择交互，支持默认值、会话覆盖值、effort 联动和快捷键调节。
2. 选择结果回写 `mainLoopModel/mainLoopModelForSession`，并按策略持久化 effort。
3. 对 fast mode、模型能力与显示文案的耦合有显式判断分支。

## 3.3 `commands/model/model.tsx`

已核对要点：

1. `/model` 命令统一入口，支持查看当前模型、交互选择与直接设置。
2. 设置路径内显式校验 allowlist、1M 可用性和自定义模型合法性。
3. 模型切换后同步处理 fast mode 兼容与提示文案拼装。

## 4. 别名、白名单与远程验证

## 4.1 `utils/model/aliases.ts`

已核对要点：

1. 定义 `sonnet/opus/haiku/best/opusplan` 等 alias 与 family alias。
2. family alias 支持“全家族通配”，供 allowlist 策略复用。

## 4.2 `utils/model/modelAllowlist.ts`

已核对要点：

1. 支持三层匹配：家族别名通配、版本前缀匹配、完整 ID 精确匹配。
2. 当同一 family 既有通配又有更具体条目时，具体条目优先（防误放宽）。
3. 兼容 alias 双向解析，避免展示名与底层 ID 不一致造成误判。

## 4.3 `utils/model/validateModel.ts`

已核对要点：

1. 本地先过 allowlist 与 alias 快速校验，再按需发起 side query 验证。
2. 对 `NotFoundError`、鉴权失败、网络失败做差异化错误文案。
3. 3P provider 下提供版本回退建议链（如 4.6 回退到旧版本）。

## 5. provider 与能力门控

## 5.1 `utils/model/providers.ts`

已核对要点：

1. provider 来源于 `CLAUDE_CODE_USE_*` 环境开关，分为 firstParty/bedrock/vertex/foundry。
2. 提供 first-party base URL 判定，作为能力与缓存更新门控前置条件。

## 5.2 `utils/model/modelStrings.ts`

已核对要点：

1. 统一维护 provider -> model 字符串映射并支持 settings 覆盖。
2. bedrock 场景支持异步拉取 profile 并回填映射；默认值与最终值可阶段性不同。
3. 提供 `resolveOverriddenModel` 用于将 override 值还原到 canonical 名称。

## 5.3 `utils/model/modelCapabilities.ts`

已核对要点：

1. 模型能力缓存写入本地 `cache/model-capabilities.json`，权限模式 `0600`。
2. 仅在 ant + first-party + 合法 base URL 场景启用拉取，避免跨 provider 误读。
3. 匹配策略“最长ID优先 + 稳定排序”，提高子串匹配准确性。

## 5.4 `utils/model/modelSupportOverrides.ts`

已核对要点：

1. 3P provider 可通过环境变量声明能力覆盖（effort/thinking 等）。
2. 覆盖按模型精确匹配，不命中时返回 `undefined`，不污染默认能力判定。

## 5.5 `utils/model/check1mAccess.ts`

已核对要点：

1. 1M 可用性由全局禁用开关 + extra usage provision 状态联合决定。
2. 订阅用户与非订阅用户走不同判定路径，统一暴露 `checkOpus1mAccess/checkSonnet1mAccess`。

## 6. 运行时一致性补偿

## 6.1 `hooks/useMainLoopModel.ts`

已核对要点：

1. 通过订阅 growthbook 刷新信号强制重渲染，修复“显示模型与执行模型短暂不一致”问题。
2. 统一从 session override / app state / default setting 三层解析最终模型。

## 7. 子系统不变量

1. 模型解析必须保持“会话覆盖优先于持久设置”的顺序不变。
2. allowlist 判定必须先于远程验证，避免无效模型触发额外网络请求。
3. 3P provider 的 model string/能力覆盖必须与 canonical 名称映射保持可逆。
4. 1M 能力判定必须由统一函数出口控制，避免 UI 与命令路径出现分歧。

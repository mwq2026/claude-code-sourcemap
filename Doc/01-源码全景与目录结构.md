# 01｜源码全景与目录结构

## 1. 仓库顶层结构

顶层可见内容：

- `README.md`：说明该项目是基于 npm 包和 source map 还原，研究用途
- `extract-sources.js`：提取 `cli.js.map` 中 `sourcesContent` 的脚本
- `package/`：发布产物目录（含 `package.json`、`cli.js`、`cli.js.map` 等）
- `restored-src/`：还原源码目录（本次主要分析对象）

## 2. 源码总体规模（`restored-src/src`）

统计结论（脚本扫描结果）：

- 源文件总数：约 **1902**
- 一级目录（含根文件）覆盖：命令、工具、服务、状态、UI、插件、技能、远程等完整链路

按一级分区（节选）：

- `utils/`：564 files，180487 lines（最大公共能力层）
- `components/`：389 files，81892 lines（Ink/React UI 视图层）
- `tools/`：184 files，50863 lines（工具实现层）
- `commands/`：207 files，26528 lines（命令编排层）
- `services/`：130 files，53683 lines（后端能力/协议层）
- `hooks/`：104 files，19232 lines（交互与状态桥接）

这说明项目是“**重工具链 + 重运行时编排 + 重终端交互**”的 CLI Agent 系统。

## 3. 顶层文件角色（`src/*.ts(x)`）

关键顶层文件：

- `main.tsx`：CLI 入口与总调度中心
- `query.ts`：对话主循环（message/tool_use/tool_result 迭代）
- `QueryEngine.ts`：会话级引擎封装（SDK/无头路径）
- `commands.ts`：命令注册表与启用策略
- `tools.ts`：工具注册表与权限前置过滤
- `Tool.ts`：工具协议与上下文类型定义
- `setup.ts` / `history.ts` / `context.ts`：会话初始化、历史、上下文注入
- `interactiveHelpers.tsx` / `replLauncher.tsx`：Ink 渲染与 REPL 挂载

## 4. 一级目录职责图

### 4.1 交互与入口

- `entrypoints/`：启动初始化与 SDK 入口类型
- `screens/`：REPL 等主屏
- `ink/`：终端渲染底座

### 4.2 任务执行核心

- `query/`：query 循环中的配置、状态转移、预算策略
- `tools/`：Bash、Read/Edit、Web、MCP、Agent、Task 等工具实现
- `commands/`：`/commit` `/review` `/config` `/mcp` `/skills` 等命令

### 4.3 能力服务

- `services/api/`：Claude API 封装、重试、日志、bootstrap、files API
- `services/mcp/`：MCP 客户端、配置、鉴权、连接管理
- `services/analytics/`：埋点、实验开关（GrowthBook）、日志上报

### 4.4 状态与上下文

- `bootstrap/state.ts`：跨会话生命周期全局状态容器（高度关键）

- `context/`：通知/统计/弹层/语音等 React Context

### 4.5 扩展生态

- `skills/`：技能目录扫描、frontmatter 解析、MCP 技能桥接
- `plugins/` + `services/plugins/`：内建插件与插件加载
- `remote/` + `bridge/`：远程会话、WebSocket、桥接协议

### 4.6 专项模式

- `assistant/`：Assistant（KAIROS）路径

- `voice/`：语音模式开关

## 5. 代码风格与工程特征

### 5.1 大量 feature gate（`feature('...')`）

项目通过 `bun:bundle` 的 `feature()` 做编译期/构建期裁剪：

- `KAIROS`、`COORDINATOR_MODE`、`VOICE_MODE`、`HISTORY_SNIP` 等
- 常见写法：按条件 `require(...)`，避免无用模块进入构建产物

### 5.2 明确的“延迟加载 + 去环依赖”策略

- 入口和核心层出现大量注释说明“为什么要 lazy import / require”
- 典型目的：
  1. 减少冷启动
  2. 打破循环依赖（例如主入口与 UI/状态互相引用）
  3. 配合 feature gate 做 dead code elimination

### 5.3 强类型与协议边界

- `Tool.ts`、`types/*`、`entrypoints/*` 维持工具、命令、消息、权限协议统一
- 通过 `zod` 与 schema 做运行时校验（尤其 API 响应和工具输入）

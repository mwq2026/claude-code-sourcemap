# 12｜文件级索引：命令与工具（细颗粒度）

> 目标：把“命令系统 + 工具系统”落到文件级职责，便于学习者快速定位，也便于研发改动时做影响面检查。  
> 口径：仅记录当前源码可核验实现。

---

## 1. 命令系统文件级索引（`restored-src/src/commands.ts`）

## 1.1 聚合与装配逻辑（已核对）

- `commands.ts` 直接 import 大量内建命令，并通过 `feature()` / `process.env` 条件引入可裁剪命令。
- `INTERNAL_ONLY_COMMANDS` 集中定义 ant-only/internal 命令清单。
- `COMMANDS` 通过 `memoize(() => [...])` 组装稳定命令数组。
- `getCommands(cwd)`：
  - 先加载 skills/plugin/workflow/builtin 全部来源
  - 再做 `meetsAvailabilityRequirement` 与 `isCommandEnabled` 动态过滤
  - 再把动态 skills 插入到 builtin 前面（保持交互体验与优先级）

---

## 1.2 代表性命令文件职责（抽样）

### A. `commands/review.ts`

- `review`：`type: 'prompt'`，本地 prompt 驱动 PR review。
- `ultrareview`：`type: 'local-jsx'`，描述中明确“Claude Code on the web”路径，受 `isUltrareviewEnabled()` 控制。
- 关键价值：把“本地审查”和“web 远程审查”拆成两个清晰入口，避免语义混淆。

### B. `commands/compact/index.ts`

- 命令名：`compact`
- 类型：`local`
- 行为：`/compact` 清理历史并保留总结，可接收可选 summarization 指令
- 开关：`DISABLE_COMPACT` 环境变量可禁用
- 标记：`supportsNonInteractive: true`

---

## 1.3 命令可用性过滤（关键判断点）

`meetsAvailabilityRequirement(cmd)` 的核心分支：

1. `claude-ai`：仅 Claude.ai 订阅用户
2. `console`：控制台/API key 场景，排除 3P 与自定义网关路径

研发意义：

- 命令“是否展示”不只由 feature 决定，还与账号/provider 绑定，改命令时需检查 availability。

## 2. 工具系统文件级索引（`restored-src/src/tools.ts` + `Tool.ts`）

## 2.1 工具池装配（已核对）

- `getAllBaseTools()`：内建工具全集来源（含 feature/env 条件）
- `getTools(permissionContext)`：
  - simple mode 下返回缩减工具集（含 coordinator 特殊分支）
  - REPL mode 下隐藏 `REPL_ONLY_TOOLS` 原语工具
  - 始终走 deny 规则过滤
- `assembleToolPool(permissionContext, mcpTools)`：
  - built-in + MCP 合并
  - 分区排序后 `uniqBy(name)`，保 built-in 前缀稳定（缓存友好）
- `getMergedTools()`：直接并集（用于 token 统计等）

---

## 2.2 Tool 协议核心（`Tool.ts`）

`ToolUseContext` 已核对关键字段：

1. 资源与配置：`commands/tools/mcpClients/mcpResources/agentDefinitions`
2. 状态读写：`getAppState/setAppState/setAppStateForTasks`
3. 交互与通知：`setToolJSX/addNotification/sendOSNotification`
4. 中断与运行时：`abortController/setInProgressToolUseIDs/setHasInterruptibleToolInProgress`
5. 追踪：`queryTracking/toolDecisions/toolUseId`

研发意义：

- 新工具若要影响 UI、状态、权限、中断，几乎都要通过 `ToolUseContext` 这条边界。

---

## 2.3 代表性工具文件职责（抽样）

### A. `tools/BashTool/BashTool.tsx`

已核对实现要点：

1. 输入 schema（`command/timeout/description/run_in_background/...`）
2. 永久隐藏内部字段 `_simulatedSedEdit`（防绕过权限与 sandbox）
3. 解析并识别 read/search/list/silent 命令语义（影响 UI 展示与摘要）
4. 支持 background task 语义与自动后台策略
5. 与权限、sandbox、文件历史、task 输出、进度事件深度耦合

### B. `tools/FileEditTool/FileEditTool.ts`

已核对实现要点：

1. `buildTool` 定义 `name/searchHint/maxResultSize/strict`
2. `checkPermissions` 走 `checkWritePermissionForTool`
3. `backfillObservableInput` 对 `file_path` 做 `expandPath`
4. 安全与稳健校验：
   - team memory secret guard
   - deny rule 匹配
   - UNC 路径特殊防护（避免 NTLM credential leak）
   - 文件大小上限（1 GiB）防 OOM
   - 旧内容/新内容一致性、不存在文件提示、ipynb 工具分流

### C. `tools/AgentTool/AgentTool.tsx`

已核对实现要点：

1. 复杂 schema 由 feature/gate 动态裁剪（如 `cwd/run_in_background/isolation`）
2. prompt 构造阶段会按 MCP server 可用性与权限规则过滤 agent
3. 同时支持同步结果与异步/远程/teammate 等内部状态结果
4. 与任务系统、远程任务、worktree、teleport、progress 跟踪深度集成

---

## 2.4 工具权限入口（`hooks/useCanUseTool.tsx`）

已核对决策路径：

1. `hasPermissionsToUseTool` 先给出 allow/deny/ask
2. allow：可直接放行并记录决策来源
3. deny：直接拒绝，并可记录 auto-mode denial
4. ask：
   - coordinator path（可先等自动检查）
   - swarm worker path
   - interactive path（入队等待用户确认）
5. 任何异常/中断最终都会走取消与清理（含 classifier checking 清理）

研发意义：

- 工具权限不是单一弹窗逻辑，而是多运行模式统一入口；改权限行为须全链路回归。

---

## 3. 命令与工具的耦合边界（实践清单）

当改动命令/工具时，建议至少核对：

1. 是否破坏 `commands.ts` 装配顺序与可用性过滤
2. 是否破坏 `tools.ts` 的 deny 过滤与 REPL/simple 模式分支
3. 工具 schema 是否泄露内部字段（尤其可绕过权限的字段）
4. 工具中断/失败是否仍能形成完整 `tool_result` 闭环
5. 影响到 Agent/Bash/FileEdit 的改动是否同步检查任务、权限、远程路径

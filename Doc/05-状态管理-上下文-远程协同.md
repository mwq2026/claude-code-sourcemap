# 05｜状态管理、上下文系统与远程协同

## 1. 双层状态架构

项目状态大体分两层：

1. **会话级全局状态**：`bootstrap/state.ts`
2. **UI/运行态状态**：`state/AppStateStore.ts` + `AppState.tsx`

## 2. `bootstrap/state.ts`：全局会话事实中心

该文件集中维护大量 session 级信息（并有注释强调“谨慎新增”）：

- 基础运行：`cwd`、`projectRoot`、`sessionId`
- 计量指标：成本、token、duration、工具统计
- 鉴权与配置：ingress token、settings source、api key fd
- telemetry 对象：meter/provider/logger/tracer
- 特性状态：remote mode、kairos、worktree、prompt cache latch
- 运行痕迹：lastAPIRequest、lastMainRequestId、错误日志

它是整个运行时的“共享事实仓库”，承担跨模块数据桥接职责。

## 3. `AppStateStore.ts`：交互态状态中心

`AppState`（强类型定义）覆盖：

- 界面行为：footer 选择、expanded view、status line
- 权限上下文：`toolPermissionContext`
- 远程桥接状态：连接、重连、session URL、错误信息
- MCP 与插件当前装载态
- 任务列表、前台任务、agent name registry
- 通知队列、elicitation 队列
- prompt suggestion / thinking 开关

这说明 UI 并不是“薄壳”，而是一个完整的事件驱动状态机。

## 4. 上下文系统：`context.ts` + `context/*`

### 4.1 模型上下文（prompt 注入）

`context.ts` 提供两类核心上下文：

- `getSystemContext()`：git 快照、cache breaker 等
- `getUserContext()`：CLAUDE.md 与日期信息

均使用 memoize 缓存，并在注入字段变化时主动清缓存。

### 4.2 React Context（UI 运行时）

`context/` 目录包含：

- 通知（notifications）
- 统计（stats）
- 模态与弹层（modal/overlay/promptOverlay）
- 语音等状态容器

将跨组件共享状态从业务逻辑里抽离，减少 prop drilling。

## 5. Bootstrap API 缓存：`services/api/bootstrap.ts`

该模块用于获取并缓存客户端启动数据（如模型选项）：

- 首先检查隐私策略与 API provider
- OAuth 与 API key 双通道鉴权
- `zod` 校验响应结构，失败直接降级
- 仅当内容变更时落盘（`isEqual` 对比）避免频繁写配置

这是典型“**远端配置拉取 + 本地稳定缓存**”模式。

## 6. 远程协同会话：`RemoteSessionManager`

与 `SessionsWebSocket`、`teleport/api` 协同实现：

- 消息接收与派发（SDK message 与 control message 分流）
- 权限请求生命周期（发起、取消、响应）
- 断连重连通知
- 本地发送消息到远端 session

它使 CLI 不只是单机工具，而是可承载远程代理会话的终端前端。

## 7. 为什么这种状态拆分有效

1. **bootstrap state**：适合“全局事实 + 生命周期长 + 跨层共享”
2. **app state**：适合“UI响应式 + 高频更新 + 局部订阅”
3. **context 层**：适合“横切关注点（通知/弹层/统计）”

这种拆分减少了单一大状态对象的复杂耦合，也提高了并行开发可行性。

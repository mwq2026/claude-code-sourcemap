# 05｜状态管理、上下文系统与远程协同

## 1. 双层状态架构

项目状态大体分两层：

1. **会话级全局状态**：`bootstrap/state.ts`
2. **UI/运行态状态**：`state/AppStateStore.ts` + `AppState.tsx`

## 2. `bootstrap/state.ts`：全局会话事实中心

该文件集中维护大量 session 级信息（并有注释强调“谨慎新增”）：

- 基础运行：`cwd`、`projectRoot`、`sessionId`
- 计量指标：成本、token、duration、工具统计
- 鉴权与配置：ingress token、settings source、api key fd
- telemetry 对象：meter/provider/logger/tracer
- 特性状态：remote mode、kairos、worktree、prompt cache latch
- 运行痕迹：lastAPIRequest、lastMainRequestId、错误日志

它是整个运行时的“共享事实仓库”，承担跨模块数据桥接职责。

## 3. `AppStateStore.ts`：交互态状态中心

`AppState`（强类型定义）覆盖：

- 界面行为：footer 选择、expanded view、status line
- 权限上下文：`toolPermissionContext`
- 远程桥接状态：连接、重连、session URL、错误信息
- MCP 与插件当前装载态
- 任务列表、前台任务、agent name registry
- 通知队列、elicitation 队列
- prompt suggestion / thinking 开关

这说明 UI 并不是“薄壳”，而是一个完整的事件驱动状态机。

## 4. 上下文系统：`context.ts` + `context/*`

### 4.1 模型上下文（prompt 注入）

`context.ts` 提供两类核心上下文：

- `getSystemContext()`：git 快照、cache breaker 等
- `getUserContext()`：CLAUDE.md 与日期信息

均使用 memoize 缓存，并在注入字段变化时主动清缓存。

### 4.2 React Context（UI 运行时）

`context/` 目录包含：

- 通知（notifications）
- 统计（stats）
- 模态与弹层（modal/overlay/promptOverlay）
- 语音等状态容器

将跨组件共享状态从业务逻辑里抽离，减少 prop drilling。

## 5. Bootstrap API 缓存：`services/api/bootstrap.ts`

该模块用于获取并缓存客户端启动数据（如模型选项）：

- 首先检查隐私策略与 API provider
- OAuth 与 API key 双通道鉴权
- `zod` 校验响应结构，失败直接降级
- 仅当内容变更时落盘（`isEqual` 对比）避免频繁写配置

这是典型“**远端配置拉取 + 本地稳定缓存**”模式。

## 6. 远程协同会话：`RemoteSessionManager`

与 `SessionsWebSocket`、`teleport/api` 协同实现：

- 消息接收与派发（SDK message 与 control message 分流）
- 权限请求生命周期（发起、取消、响应）
- 断连重连通知
- 本地发送消息到远端 session

它使 CLI 不只是单机工具，而是可承载远程代理会话的终端前端。

### 6.1 通信分层（HTTP + WebSocket）

远程链路是“双通道”：

1. **HTTP**：发送用户事件到会话  
   - `POST /v1/sessions/{sessionId}/events`
   - 由 `sendEventToRemoteSession()` 组装 `events` 报文。
2. **WebSocket**：订阅会话消息流与控制流  
   - `wss://.../v1/sessions/ws/{sessionId}/subscribe?organization_uuid=...`
   - 由 `SessionsWebSocket` 连接并分发给 `RemoteSessionManager`。

### 6.2 HTTP 发送报文（用户消息）

`utils/teleport/api.ts` 中 `sendEventToRemoteSession()` 的报文核心结构：

```json
{
  "events": [
    {
      "uuid": "event-uuid",
      "session_id": "sess_xxx",
      "type": "user",
      "parent_tool_use_id": null,
      "message": {
        "role": "user",
        "content": "user text or content blocks"
      }
    }
  ]
}
```

字段含义：

- `uuid`： 事件唯一标识，用于幂等/去重与回显过滤。
- `session_id`： 目标远程会话 ID。
- `type`： 事件类型，此处为 `user`。
- `parent_tool_use_id`： 父工具调用关联（普通用户输入通常为 `null`）。
- `message.role`： 消息角色（`user`）。
- `message.content`： 字符串或内容块数组（与 Anthropic 消息内容结构兼容）。

### 6.3 WS 入站报文类型与解析

`RemoteSessionManager.handleMessage()` 对入站消息做分流：

1. `control_request`：控制请求（如 `can_use_tool` 权限请求）
2. `control_cancel_request`：取消某个 pending control request
3. `control_response`：控制响应确认
4. 其他：按 SDKMessage 交给 UI/上层消费

对应协议 schema 在 `entrypoints/sdk/controlSchemas.ts`。

### 6.4 控制报文字段详解（权限与中断）

**A. 权限请求（服务端 -> 客户端）**

```json
{
  "type": "control_request",
  "request_id": "req_123",
  "request": {
    "subtype": "can_use_tool",
    "tool_name": "Bash",
    "input": { "command": "ls -la" },
    "tool_use_id": "toolu_abc"
  }
}
```

- `request_id`：本次控制请求唯一 ID，响应必须回填。
- `request.subtype=can_use_tool`：请求客户端做工具权限决策。
- `tool_name/input`：候选工具与输入参数。
- `tool_use_id`：绑定具体工具调用实例。

**B. 权限响应（客户端 -> 服务端）**

```json
{
  "type": "control_response",
  "response": {
    "subtype": "success",
    "request_id": "req_123",
    "response": {
      "behavior": "allow",
      "updatedInput": { "command": "ls -la" }
    }
  }
}
```

或拒绝：

```json
{
  "type": "control_response",
  "response": {
    "subtype": "success",
    "request_id": "req_123",
    "response": {
      "behavior": "deny",
      "message": "User denied tool execution"
    }
  }
}
```

- `behavior`：`allow`/`deny`。
- `updatedInput`：允许时可返回调整后的输入。
- `message`：拒绝原因文本。

**C. 中断请求（客户端 -> 服务端）**

> 说明：该示例就是当前代码路径发送的最小完整结构（`type/request_id/request.subtype`）；业务上可叠加链路级元信息头，但不改变该消息体核心字段。

```json
{
  "type": "control_request",
  "request_id": "req_interrupt_x",
  "request": { "subtype": "interrupt" }
}
```

用于取消当前远程会话中的执行。

### 6.5 完整交互时序（远程工具调用）

```text
本地用户输入
  -> HTTP POST /sessions/{id}/events (type=user)
  -> 服务端开始推理并通过 WS 推送 SDKMessage/stream_event
  -> 若需高风险工具，服务端发 control_request(can_use_tool)
  -> 客户端弹权限 UI，回 control_response(allow/deny)
  -> 服务端继续执行并回传 assistant/tool_progress/result
  -> 本地渲染完成，进入下一轮
```

### 6.6 控制面“对账关系”与字段约束（补充）

为避免控制报文串线，可按以下规则核对：

1. `control_request.request_id` 与 `control_response.response.request_id` 必须一一对应。
2. `can_use_tool.tool_use_id` 与后续工具执行上下文必须绑定同一 tool use 实例。
3. 收到 `control_cancel_request.request_id` 后，本地应取消对应 pending 权限弹窗，不再发送成功响应。

可视化对账：

```text
control_request(req_1, tool_use_id=toolu_a)
  -> pendingPermissionRequests["req_1"] = ...
  -> 用户决策
     -> control_response(success, request_id=req_1, behavior=allow/deny)
```

若服务端先发取消：

```text
control_request(req_2)
  -> pending
control_cancel_request(req_2)
  -> pending 删除
  -> UI 取消该请求
  -> 不应再发 success(req_2)
```

### 6.7 错误/兼容分支报文（补充）

#### A. 未识别 control subtype

`RemoteSessionManager` 对未知 subtype 会主动回：

```json
{
  "type": "control_response",
  "response": {
    "subtype": "error",
    "request_id": "原 request_id",
    "error": "Unsupported control request subtype: ..."
  }
}
```

目的：避免服务端一直等待，导致控制面“悬挂”。

#### B. WebSocket 关闭码与重连语义

- `4003`：视为永久拒绝，不重连。
- `4001`：`session not found`，按有限预算重试（最多 3 次）。
- 其他瞬断：在最大重连次数内退避重连。

这解释了为什么抓包会出现“短时间多次 subscribe 连接尝试”。

### 6.8 keep_alive 与业务报文的区分

`controlSchemas` 中定义了 `keep_alive` 消息；其语义是保活连接，不代表业务状态变化。
排障时应将其与 `control_request/response`、`SDKMessage` 区分，避免误判为“空响应”。

## 7. 为什么这种状态拆分有效

1. **bootstrap state**：适合“全局事实 + 生命周期长 + 跨层共享”
2. **app state**：适合“UI响应式 + 高频更新 + 局部订阅”
3. **context 层**：适合“横切关注点（通知/弹层/统计）”

这种拆分减少了单一大状态对象的复杂耦合，也提高了并行开发可行性。

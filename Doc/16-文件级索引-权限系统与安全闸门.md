## 2. 规则模型与匹配主入口

## 2.1 `permissions.ts`

已核对要点：

1. allow/deny/ask 规则聚合来源统一纳入：`localSettings`、`userSettings`、`projectSettings`、`managedSettings`、`cliArg`、`command`、`session`。
2. 工具匹配通过 `toolMatchesRule`，支持：
   - 工具级规则（无 content）
   - MCP 服务级规则（如 `mcp__server` / `mcp__server__*`）。
3. 规则内容映射用 `getRuleByContentsForTool`，下游由工具做细粒度内容匹配（如 path/prefix/domain）。
4. 权限请求文案构造集中在 `createPermissionRequestMessage`，可携带 `hook/rule/classifier/subcommand` 等原因类型。

---

## 2.2 `permissionRuleParser.ts`

已核对要点：

1. 规则字符串形态：`Tool` 或 `Tool(content)`。
2. `escapeRuleContent/unescapeRuleContent` 对 `\` 与括号做双向转义，避免解析歧义。
3. 兼容 legacy tool name（如 `Task -> Agent`、`KillShell -> TaskStop`、`AgentOutputTool/BashOutputTool -> TaskOutput`）。
4. `Bash()` 与 `Bash(*)` 被视为工具级规则，不是 content 规则。

---

## 3. 路径权限与文件安全闸门

## 3.1 `filesystem.ts`

已核对要点：

1. 有危险文件与目录清单（如 `.gitconfig`、`.bashrc`、`.mcp.json`、`.claude`、`.git` 等）用于自动编辑安全闸门。
2. 路径比较做大小写归一化，覆盖大小写不敏感文件系统绕过风险。
3. 配置文件、会话计划文件、session memory、project temp 等内部路径有专门判定逻辑。
4. 多处显式注释说明了 path normalize / traversal / UNC 等安全意图。

---

## 3.2 `pathValidation.ts`

已核对要点：

1. `isPathAllowed` 按固定顺序执行：
   - deny rule
   - 可编辑内部路径（write/create）
   - 自动编辑安全检查（危险路径等）
   - working directory 自动放行（受模式约束）
   - 可读内部路径（read）
   - sandbox write allowlist（仅 write/create 且在工作目录外）
   - allow rule
2. `read` 与 `write/create` 采用不同 permissionType（`read` vs `edit`）。
3. sandbox 写白名单支持 `denyWithinAllow` 二次阻断，避免“父目录可写导致敏感子路径放开”。
4. `expandTilde` 只支持 `~`，不支持 `~username`。

## 4. 权限模式切换与可用性边界

## 4.1 `getNextPermissionMode.ts`

已核对要点：

1. 模式轮转集中在 `getNextPermissionMode/cyclePermissionMode`。
2. ant 用户路径与普通用户不同：`default` 分支可能直接走 `bypassPermissions/auto`，跳过 `acceptEdits/plan`。
3. `canCycleToAuto` 同时检查缓存态与实时 gate，避免 UI 切换触发非法转移。

## 5. 解释器与自动分类链

## 5.1 `permissionExplainer.ts`

已核对要点：

1. 解释器可开关，默认启用（可通过全局配置关闭）。
2. 通过 `sideQuery` + 强制 tool choice 输出结构化字段：`explanation/reasoning/risk/riskLevel`。
3. 支持抓取最近 assistant 上下文，辅助“为什么要执行该命令”解释。
4. 有独立 analytics 事件与错误分类（parse/network/unknown）。

---

## 5.2 `bashClassifier.ts`

已核对要点：

1. 当前文件是 external build stub：分类能力返回关闭态（`isClassifierPermissionsEnabled=false`）。
2. `classifyBashCommand` 在 stub 中固定不命中，原因是“feature disabled”。
3. 该文件清晰表达“能力存在接口，但在当前构建目标中禁用”的边界。

---

## 5.3 `yoloClassifier.ts`

已核对要点：

1. 包含 auto mode classifier 的 prompt 拼装、请求/响应解析、错误 dump 与调试输出。
2. 支持外部模板默认值抽取（用于 `auto-mode defaults/critique` 一类流程）。
3. 有 session 级错误转储路径与请求快照能力，便于事后排障。

---

## 6. 权限子系统不变量（从实现归纳）

1. deny 优先级高于 allow。
2. 写入路径即使在可写工作目录，也要先通过安全闸门检查。
3. 工具级规则与内容级规则分层处理，不混用。
4. 模式切换必须受 gate 与上下文状态双重约束。

## 2. 运行时分层与职责边界（研发视角）

### 2.1 启动与模式路由层

- 关键文件：`main.tsx`、`entrypoints/init.ts`
- 核心职责：
  - 预热副作用（MDM/keychain/perf checkpoint）
  - 初始化配置、网络、策略、远程设置
  - 解析运行模式（命令模式、REPL、remote/bridge、assistant/coordinator）
  - 在 feature gate 条件下装配可见能力面

架构特征：

- `main.tsx` 内大量 `feature()` + 条件 `require()`，用于发行裁剪与路径隔离
- 路由层承担了“能力编排控制器”角色，而非单纯参数解析器

### 2.2 会话执行内核层

- 关键文件：`QueryEngine.ts`、`query.ts`
- `QueryEngine`：
  - 持有会话级可变状态（`mutableMessages`、usage、权限拒绝记录、cache 状态）
  - 管理 turn 生命周期（`submitMessage`）
  - 承接 SDK/无头路径
- `query.ts`：
  - 负责消息规范化、模型流式消费、工具执行回填、压缩/恢复路径
  - 是“每轮推理 + 工具闭环”的主执行函数

边界意义：

- “会话容器（class）”与“轮次算法（function）”解耦，便于复用与测试

### 2.3 工具协议与编排层

- 关键文件：`Tool.ts`、`tools.ts`、`services/tools/toolOrchestration.ts`、`services/tools/StreamingToolExecutor.ts`
- `ToolUseContext` 承载：
  - 命令/工具/MCP 资源访问
  - AppState 读写
  - 通知/UI 回调/中断控制
  - 文件历史与归因更新
  - 执行中工具集合与响应长度等运行时指标

执行策略：

1. 批次分割（`partitionToolCalls`）：
   - 并发安全工具合并批处理
   - 非并发安全工具串行独占
2. 并发批执行后“顺序回放 contextModifier”
3. Streaming 执行器维护 `queued/executing/completed/yielded` 状态机
4. 中断语义按 tool `interruptBehavior` 区分 `cancel` / `block`
5. fallback/discard 场景用合成 `tool_result` 保证协议闭环

### 2.4 能力服务层

- 关键子域（按 `services/` 分布）：
  - `api/`（20 files）：模型调用、重试、bootstrap、files API、日志
  - `mcp/`（23 files）：连接管理、auth、配置、allowlist/permissions
  - `compact/`（11 files）：auto/micro/session memory compact
  - `analytics/`（9 files）：事件 sink、growthbook、metadata、killswitch
  - `lsp/`（7 files）：LSP 客户端/服务实例/诊断注册
  - `remoteManagedSettings/`（5 files）：远程托管配置同步与安全校验

### 2.5 状态与上下文层

- 关键文件：`bootstrap/state.ts`、`state/AppStateStore.ts`、`context.ts`、`context/*`
- 双层状态：
  - session/global facts（跨模块共享、生命周期长）
  - interactive app state（高频 UI 更新、远程连接态、队列态、通知态）

### 2.6 扩展与生态层

- `commands/`（207 files）：
  - plugin 子域 17 files
  - install-github-app 子域 14 files
- `tools/`（184 files）：

## 3. 关键运行时对象与不变量

## 3.1 消息模型不变量

在 query 循环中需持续维持：

1. `tool_use.id` 与 `tool_result.tool_use_id` 一一匹配
2. 会话消息可归一化为 API 可接受形态（`normalizeMessagesForAPI`）
3. partial stream 也要可恢复（`message_delta` 回填 stop_reason/usage）

## 3.2 预算与边界不变量

- turn 上限：`maxTurns`
- 成本上限：`maxBudgetUsd`
- task 预算：`taskBudget`
- 中断/重试不应破坏消息一致性（尤其 tool 回填和 compact 后重试）

## 3.3 执行一致性不变量

- 并发不等于乱序生效：context 修改需要保持确定顺序
- fallback/discard 不可留下“悬空 tool_use”
- 中断后必须保持下一轮可继续执行（协议完整）

## 4. Query 子系统细化（从实现细节看设计取舍）

### 4.1 压缩链路并非单一机制

`query.ts` 可见多层 compaction：

- snip（feature 条件）
- microcompact
- autocompact
- reactive compact（由 `services/compact/autoCompact.js` 动态加载相关路径）

工程意图：

1. 优先“细粒度降载”再做“摘要式压缩”
2. 让压缩与恢复可组合，而非一次性 hard truncate
3. 将 token 压力处理从“报错后兜底”前移为“常态运行策略”

### 4.2 错误恢复强调“连续性”

重点可见：

- `max_output_tokens` 路径有恢复/升级逻辑
- media/prompt-too-long 场景有重试与 compact 协作分支
- 通过注释可见曾规避“无限循环 + 过量 API 调用”问题

结论：该系统不是只追求成功率，而是追求“可控失败 + 可恢复继续”。

### 4.3 流式事件处理强调时序正确性

- `content_block_stop` 先产出内容
- `message_delta` 再补 stop_reason 与 usage 最终值
- 若消费者只看首个事件，容易误读轮次状态

## 5. 工具执行内核细化（并发/中断/回填）

### 5.1 串并行批处理策略

`toolOrchestration.ts` 核心算法：

1. 先分批（并发安全 vs 非并发安全）
2. 并发批：先执行收集，再按接收顺序回放 contextModifier
3. 串行批：每步即时修改上下文并推进

价值：兼顾吞吐与确定性，避免并发写上下文导致不可复现行为。

### 5.2 StreamingToolExecutor 状态机

状态：

- `queued` → `executing` → `completed` → `yielded`

关键机制：

- `canExecuteTool()`：并发窗口判断
- `discard()`：fallback 丢弃策略
- `getAbortReason()`：区分 sibling_error / user_interrupted / streaming_fallback
- `interruptBehavior`：不同工具可声明中断策略

价值：把“执行层复杂性”封装在状态机中，减少 query 主循环复杂度外泄。

### 5.3 Bash 并行失败联动

从实现注释可见，Bash 相关并发工具错误会触发 sibling abort，避免“失败后兄弟进程继续跑完”的资源浪费和副作用扩大。

## 6. 远程与控制面（Remote/Bridge/SDK）

### 6.1 双通道通信

- HTTP：`/v1/sessions/{sessionId}/events`（发送事件）
- WebSocket：`/v1/sessions/ws/{sessionId}/subscribe`（接收会话流）

### 6.2 控制消息协议

`entrypoints/sdk/controlSchemas.ts` 里定义 control 协议：

- `control_request`
- `control_response`
- `control_cancel_request`
- `keep_alive`

高频子类型：

- `can_use_tool`
- `interrupt`
- `set_permission_mode`
- `mcp_*` 系列
- `reload_plugins`

### 6.3 对账键体系（排障核心）

建议统一按以下键追踪：

1. `request_id`（控制面请求-响应配对）
2. `tool_use_id`（工具调用闭环）
3. `uuid`（事件幂等与回显过滤）
4. `session_id`（远程会话边界）
5. `task_id`（异步任务链路）

## 7. 安全与信任边界设计

### 7.1 两阶段环境变量应用

- 初始化只应用 safe env
- 信任确认后再应用完整 env

防护目标：避免在未建立信任前加载潜在高风险配置。

### 7.2 工具权限“前置过滤 + 交互决策”

- 模型看到工具前先做 deny 规则过滤（减少无效推理）
- 执行时再经 `canUseTool` 做上下文化决策
- 远程场景通过 control protocol 对齐权限流程

### 7.3 路径与执行安全

从工具目录结构可见 Bash/PowerShell/FileEdit 等工具均拆分出验证模块，体现“高风险能力模块化防护”。

## 8. 性能工程观察

1. **启动期并行预热**：MDM/keychain/API preconnect
2. **大模块延迟加载**：降低冷启动阻塞
3. **并发工具执行上限**：`CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY`（默认 10）
4. **流式优先 + 非流式 fallback**：兼顾交互体验与稳态成功率
5. **按需 compact**：控制上下文成本与失败概率

## 9. 维护与扩展建议（面向研发）

1. 新增工具必须明确：
   - `inputSchema` 校验
   - `isConcurrencySafe`
   - `interruptBehavior`
   - 权限策略与错误回填语义
2. 新增 control subtype 必须同步：
   - schema
   - 处理分支
   - cancel/error 对账行为
3. 涉及 query 重试逻辑改动时，优先验证：
   - 是否可能形成无限重试环
   - 是否会产生 orphan tool_result
4. feature gate 新增后应记录：
   - 默认值
   - 受影响路径
   - 与既有 gate 的组合影响

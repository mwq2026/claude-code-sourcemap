# 11｜协议字段与排障手册（API / SDK / Remote / Control）

> 目标读者：学习者（理解机制）+ 研发（排障定位）  
> 目标：提供字段级“速查 + 对账 + 异常定位”能力。

---

## 1. 协议视角总览

系统可分四类协议面：

1. **模型请求协议**：CLI/SDK → Anthropic Messages API
2. **模型流式事件协议**：Anthropic streaming event → 内部 message/event
3. **工具回填协议**：assistant `tool_use` ↔ user `tool_result`
4. **远程控制协议**：`control_request/response/cancel` + remote session events

---

## 2. 模型请求字段（`services/api/claude.ts`）

## 2.1 常驻核心字段

- `model`
- `messages`
- `system`
- `tools`
- `tool_choice`
- `metadata`
- `max_tokens`

## 2.2 条件字段（满足条件才发送）

- `betas`
- `thinking`
- `temperature`（thinking 关闭时）
- `context_management`
- `output_config`
- `speed`
- provider extra body params

## 2.3 字段对账建议

1. 先确认 `messages` 是否已 normalize，且 tool pairing 完整
2. 再确认 `tools/tool_choice` 与当前可见工具集合一致
3. 再确认 `max_tokens/thinking/output_config` 是否被 feature 或策略覆盖

---

## 3. 流式事件字段（`message_start` → `message_stop`）

## 3.1 关键事件序列

1. `message_start`
2. `content_block_start`
3. `content_block_delta`
4. `content_block_stop`
5. `message_delta`
6. `message_stop`

## 3.2 常见误区

1. **误把 `content_block_stop` 当最终消息完成点**  
   实际 `usage/stop_reason` 常在 `message_delta` 才稳定。

2. **误以为 tool input 一次到齐**  
   实际经 `input_json_delta.partial_json` 分片拼接。

3. **忽略内部 stream_event 透传**  
   内部常把 raw event 包装为 `stream_event` 再分发。

## 3.3 排障速查

- TTFT 异常：看 `message_start` 到首个有效内容时间
- stop_reason 异常：看 `message_delta` 是否到达
- 工具参数异常：检查 `input_json_delta` 拼接结果

---

## 4. 工具闭环协议（`tool_use` ↔ `tool_result`）

## 4.1 必守约束

1. 每个 `tool_use.id` 都应有匹配的 `tool_result.tool_use_id`
2. `tool_result` 的错误/成功语义要能驱动下一轮模型决策
3. fallback/中断场景也要保持闭环（允许合成错误结果）

## 4.2 关键故障模式

1. **orphan tool_use**：只有 `tool_use` 无 `tool_result`
2. **ID 错配**：`tool_result.tool_use_id` 指向错误调用
3. **重复回填**：同一 `tool_use_id` 多个冲突结果
4. **顺序错乱**：并发执行结果跨轮或跨消息污染

## 4.3 防护点（实现可见）

- tool orchestration 的顺序回放策略
- streaming executor 的状态机与 discard 机制
- query 重试时对旧 executor 结果隔离

## 5. 控制协议字段（`entrypoints/sdk/controlSchemas.ts`）

## 5.1 控制消息壳

- `control_request`
- `control_response`
- `control_cancel_request`
- `keep_alive`

## 5.2 高频 request subtype

- `can_use_tool`
- `interrupt`
- `set_permission_mode`
- `set_model`
- `mcp_message` / `mcp_set_servers` / `mcp_toggle` / `mcp_reconnect`
- `reload_plugins`
- `stop_task`
- `elicitation`

## 5.3 对账规则

1. `control_request.request_id` 与 `control_response.response.request_id` 一一对应
2. 收到 `control_cancel_request` 后，应终止对应 pending 决策链
3. `can_use_tool.tool_use_id` 必须关联同一工具调用实例

---

## 6. Remote 会话协议（`remote/*` + `utils/teleport/api.ts`）

## 6.1 双通道模型

1. HTTP `POST /v1/sessions/{id}/events`
2. WebSocket `/v1/sessions/ws/{id}/subscribe`

## 6.2 常见消息族

- user/assistant/result/stream_event/tool_progress/system:* 等 SDKMessage
- control_* 控制面消息

## 6.3 重连与关闭码语义（实现可见）

- 特定关闭码为永久拒绝，不重连
- session not found 有有限预算重试
- 其他瞬断走退避重连

## 7. 排障流程模板（建议直接复用）

### 7.1 “模型未按预期调用工具”

1. 看请求体 `tools/tool_choice` 是否正确注入
2. 看流中是否出现 `tool_use` block
3. 看权限是否在 `can_use_tool` 被拒绝
4. 看是否被 blanket deny 预过滤

### 7.2 “工具执行后会话卡住/异常”

1. 对账 `tool_use_id` 与 `tool_result`
2. 检查 executor 状态（queued/executing/completed/yielded）
3. 检查是否触发 discard/fallback/interruption
4. 检查下一轮 API 调用是否收到正确回填消息

### 7.3 “远程会话权限弹窗异常”

1. 看 `control_request` 是否到达客户端
2. 看 pending 队列是否被 `cancel_request` 提前清理
3. 看 `control_response` request_id 是否正确
4. 看服务端后续是否继续推流

### 7.4 “上下文超长/重试循环”

1. 查 `prompt-too-long` 或 `max_output_tokens` 错误路径
2. 查 micro/auto/reactive compact 是否触发
3. 查重试 transition reason 是否重复同一路径
4. 防止形成“compact->error->compact”死循环

---

## 8. 研发改动前后的协议检查清单

每次改动 query/tool/remote/control 相关逻辑后建议检查：

1. schema 是否同步更新
2. 关键 ID 对账链是否保持
3. 异常路径是否仍有闭环消息
4. 中断/重连/cancel 是否可恢复
5. feature gate 关闭时是否不泄露路径依赖

---

## 9. 学习者实战练习（协议视角）

1. 自行绘制一次完整轮次的“事件时间线”  
   `user -> api request -> stream events -> tool_use -> tool_result -> next request`

2. 任选一个工具（如 BashTool）绘制“权限决策状态图”  
   `request -> allow/deny -> execute/cancel -> result`

3. 远程链路绘制“控制面对账图”  
   `control_request(req_x) -> pending -> response(req_x) / cancel(req_x)`

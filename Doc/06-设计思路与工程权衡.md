# 06｜设计思路与工程权衡

## 1. 总体架构意图

该项目并非“普通 CLI 命令集合”，而是一个：

- 具备长会话记忆与多轮推理能力的 Agent Runtime
- 支持丰富工具调用（文件、终端、网络、MCP、子代理）
- 支持 UI 交互（Ink）与无头 SDK 路径并行
- 支持插件、技能、远程会话、特性门控多发行模式

这决定了它必须在以下矛盾中平衡：

1. 启动性能 vs 功能完整性  
2. 动态扩展 vs 类型安全  
3. 并发执行 vs 顺序一致性  
4. 自动化能力 vs 权限/信任边界  
5. 多形态发行 vs 单代码库维护成本  

## 2. 关键设计选择与收益

### 2.1 feature gate + 条件 require

表现：

- `feature('KAIROS')` / `feature('COORDINATOR_MODE')` / `feature('VOICE_MODE')`
- gate 内动态 `require` 模块

收益：

- 不同发行形态可共享代码树
- 无关模块可被构建裁剪，减小体积/冷启动开销
- 实验功能可快速灰度

代价：

- 导入关系更复杂，阅读门槛上升
- gate 组合增多会提高测试矩阵复杂度

### 2.2 延迟加载 + 去循环依赖

表现：

- `main.tsx` 和 `tools.ts` 中多处 lazy require 注释
- 大模块（UI、insights 等）按需 import

收益：

- 冷启动速度更可控
- 打破复杂模块互相引用导致的环依赖

代价：

- 静态分析难度更高
- 错误可能后移到运行时触发

### 2.3 统一协议层（Tool/Command/Message）

表现：

- `Tool.ts` 中上下文、权限、进度、状态约束
- `types/*` 与 `entrypoints/*` 细化接口

收益：

- 工具、命令、SDK、REPL 共享统一语义
- 重构时边界清晰，扩展成本可控

代价：

- 类型体积大，上手需要时间

### 2.4 工具执行并发策略

表现：

- `isConcurrencySafe` 决策并发批次
- 并发执行但 contextModifier 顺序回放

收益：

- 充分利用并发提升吞吐
- 保留上下文状态演化的可预测性

代价：

- 调度代码复杂，需要严谨测试中断/失败路径

### 2.5 安全边界设计（Trust + Permission）

表现：

- onboarding/trust 与 env 应用分阶段
- tool 使用前有 permissionContext 与 deny 规则过滤
- 非交互/远程模式下权限与 UI 处理路径分化

收益：

- 自动化能力不会直接越过用户信任边界
- 风险能力可控、可审计

代价：

- 用户体验与安全弹窗之间需持续调优

## 3. 数据流与控制流的“分层闭环”

典型闭环：

1. 用户输入（命令/REPL）
2. 上下文组装（system/user/memory）
3. 模型调用（streaming）
4. 工具执行（串并行调度 + 权限控制）
5. 结果回填（tool_result）
6. 状态落地（session/history/cost）
7. UI 呈现与下一轮等待

该闭环在 REPL、SDK、remote 三种模式下复用核心逻辑，仅在入口与 I/O 桥接层差异化。

## 4. 为什么“看起来复杂但合理”

从工程视角看，复杂度来自真实需求叠加：

- 需要像 IDE 助手一样长期会话
- 需要像终端工具一样高性能启动
- 需要像平台产品一样支持插件/市场/远程
- 需要像企业软件一样考虑策略、审计、权限

因此，项目选择了“强分层 + 强类型 + 强特性门控”的路线，这是高复杂 Agent CLI 的常见演化结果。

## 5. 对阅读源码的建议路径（实践）

建议阅读顺序：

1. `main.tsx`（入口路由）
2. `entrypoints/init.ts`（初始化）
3. `interactiveHelpers.tsx`、`replLauncher.tsx`（交互壳）
4. `query.ts`、`QueryEngine.ts`（主循环）
5. `Tool.ts` + `tools.ts` + `toolOrchestration.ts`
6. `commands.ts` + 典型命令（review/commit/config/mcp）
7. `bootstrap/state.ts` + `state/AppStateStore.ts`
8. `skills/`、`plugins/`、`remote/`

这样能最快建立“主干优先，再分叉细化”的心智图。

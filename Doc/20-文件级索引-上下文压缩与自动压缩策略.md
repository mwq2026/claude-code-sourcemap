## 2. `compact.ts`（主动/手动压缩主链）

已核对要点：

1. 提供对话压缩主流程，含 pre/post compact hooks、streaming 重试与 PTL 回退分支。
2. 包含对 message 内容的压缩前清洗（如 image/document 替换标记、可重注入 attachment 过滤）。
3. 定义了 post-compact 恢复预算与上限（文件、技能、token 预算）。
4. 与 queryModelWithStreaming、promptCacheBreakDetection、session/activity 等子系统耦合。

---

## 3. `microCompact.ts`（细粒度工具结果清理）

已核对要点：

1. 定义可清理工具集合（Read/Grep/Glob/WebFetch/WebSearch/FileEdit/FileWrite/Shell 系）。
2. 提供 token 估算与 tool_result 大小评估，用于触发微压缩策略。
3. 支持 cached microcompact 状态：pending edits / pinned edits / tools-sent 状态管理。
4. time-based microcompact 与 cached microcompact 有优先级与短路关系。
5. 输出含 `TIME_BASED_MC_CLEARED_MESSAGE` 标记，保证上下游可识别“旧结果已清理”。

---

## 4. `autoCompact.ts`（阈值判断与自动触发）

已核对要点：

1. 计算有效上下文窗口：context window - summary 预留输出 token。
2. 提供 warning/error/autocompact/blocking 多阈值判定函数。
3. auto-compact 支持配置与环境变量开关，且有递归/场景防护（如 session_memory/compact 等）。
4. 引入 consecutive failures 断路器，避免失败后无穷重试。
5. 与 session memory compaction、post compact cleanup 等流程协同。

---

## 5. 压缩子系统不变量

1. 微压缩优先处理高噪声工具结果，不改动核心语义消息。
2. 自动压缩触发必须受有效窗口与模式开关双重约束。
3. 压缩失败要可恢复并可观察，不能静默吞掉上下文风险。

---

## 6. 改动检查清单（compact）

1. 改阈值：是否同步 warning/error/blocking 判断一致性
2. 改微压缩：是否保持可清理工具集合与清理标记兼容
3. 改压缩重试：是否仍有断路器防止失败风暴

## 2. 核心执行层

## 2.1 `services/tools/toolExecution.ts`

已核对要点：

1. 提供单次 tool_use 执行主路径，贯穿权限判定、错误归类、消息生成与 telemetry 上报。
2. 包含 decision reason 到 OTel source 的映射逻辑，保持权限来源可观测性。
3. 支持 MCP 工具元信息抽取、hook 执行前后处理及失败路径分流。
4. 对错误类型有专门分类函数（含 telemetry-safe 约束），减少混淆日志。

---

## 2.2 `services/tools/toolHooks.ts`

已核对要点：

1. 封装 `PreToolUse/PostToolUse/PostToolUseFailure` hook 协同调用。
2. hook 返回的 blocking/cancel/additional-context 会转成标准 attachment 消息。
3. 对 hook 执行中的取消与异常有单独事件与降级路径。
4. 支持 MCP 输出在 hook 后被替换（`updatedMCPToolOutput`）。

---

## 3. 批处理编排层

## 3.1 `services/tools/toolOrchestration.ts`

已核对要点：

1. 负责将多个 tool_use 分批，按并发安全性做并行/串行执行。
2. `partitionToolCalls` 基于 `tool.isConcurrencySafe(input)` 做分组，异常时保守降为非并发。
3. 并发批次会暂存 contextModifier，按工具顺序回放到上下文，保证一致性。
4. 串行批次每步即时更新 context，保持顺序语义。

---

## 3.2 `services/tools/StreamingToolExecutor.ts`

已核对要点：

1. 面向流式 tool_use 到达场景，支持“可并发工具并行 + 非并发工具独占”。
2. 内部追踪 `queued/executing/completed/yielded` 状态与 in-progress IDs。
3. 有 sibling abort 机制：某些失败（尤其并行 Bash 场景）可触发兄弟任务快速中止。
4. 支持 streaming fallback/discard，丢弃失效推测执行结果并返回合成错误消息。
5. 中断行为与 `interruptBehavior(cancel/block)` 协同，避免错误取消语义。

---

## 4. 子系统不变量

1. 并发安全性判定失败时必须回落为保守串行。
2. context 修改应用顺序应与 tool_use 原始顺序一致。
3. hook 异常不得中断整条主链，应降级为可观测 attachment/事件。
4. 流式 fallback 后不得继续消费已失效执行结果。

---

## 5. 改动检查清单（tools service）

1. 改并发分组：是否影响串并行边界与顺序一致性
2. 改 hook 处理：是否仍保证 blocking/cancel/additional-context 语义完整
3. 改流式执行：是否保持 sibling abort 与 discard/fallback 行为一致
4. 改 telemetry 映射：是否破坏权限来源与错误类型可观测性

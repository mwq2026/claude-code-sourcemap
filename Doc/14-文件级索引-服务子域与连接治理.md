## 1. services 总体分层（本卷聚焦）

本卷按“高耦合运行路径”拆为五组：

1. MCP 连接与配置（`services/mcp/*`）
2. API 重试与启动预热（`services/api/*`）
3. 压缩策略（`services/compact/*`）
4. LSP 生命周期（`services/lsp/*`）
5. Analytics 事件总线（`services/analytics/*`）

---

## 2. MCP 子域：连接、配置、权限中继

## 2.1 `services/mcp/client.ts`（连接与调用核心）

已核对要点：

1. 同时支持 `stdio/sse/streamableHttp/ws/sdk-control` 等传输接入。
2. 统一构建 MCP tool/command/resource，并向上层提供连接结果。
3. 错误模型有专门类型：
   - `McpAuthError`
   - `McpSessionExpiredError`
   - `McpToolCallError_*`
4. `isMcpSessionExpiredError` 用 “HTTP 404 + JSON-RPC -32001” 双信号识别会话失效，避免把普通 404 误判成 session 过期。
5. 存在认证缓存路径（`mcp-needs-auth-cache.json`）与 TTL 机制，避免重复 auth 抖动。

---

## 2.2 `services/mcp/config.ts`（配置汇聚与去重）

已核对要点：

1. `getEnterpriseMcpFilePath()` 指向托管配置文件。
2. `.mcp.json` 写入使用“临时文件 + datasync + rename”原子替换，且保留原文件权限。
3. `getMcpServerSignature()` 对 server 计算内容签名：
   - stdio 取 command+args
   - remote 取 url（含代理 URL 反解）
4. `dedupPluginMcpServers()`：manual 优先于 plugin，plugin 内部“先加载先赢”。
5. `unwrapCcrProxyUrl()` 处理 CCR 代理 URL 到原始 vendor URL 的映射，保证签名去重正确。

---

## 2.3 `services/mcp/useManageMCPConnections.ts`（连接编排）

已核对要点：

1. Hook 负责连接初始化、状态更新、重连与通知联动。
2. 维护批量更新窗口（`MCP_BATCH_FLUSH_MS=16`），把 clients/tools/commands/resources 合并后一次写入 AppState，降低频繁 setState 抖动。
3. 支持 `reconnect` 退避参数（`MAX_RECONNECT_ATTEMPTS`、`INITIAL_BACKOFF_MS`、`MAX_BACKOFF_MS`）。
4. 处理 channel permission relay（受 gate 控制）并把 callbacks 放进 AppState，供交互权限链消费。
5. 连接成功后会注册 elicitation handler，连接失败/禁用时会清理对应 mcp 工具与资源。

---

## 2.4 `services/mcp/channelPermissions.ts`（渠道权限中继）

已核对要点：

1. 中继能力由 `isChannelPermissionRelayEnabled()` 受 GrowthBook gate 控制。
2. 回复格式正则 `PERMISSION_REPLY_RE` 明确限定。
3. `shortRequestId(toolUseID)` 使用固定 alphabet 与 hash 生成 5 位短 ID，并带敏感子串规避。
4. `filterPermissionRelayClients()` 只有“已连接 + allowlist 命中 + 双 capability 声明”才可作为权限中继面。
5. `createChannelPermissionCallbacks()` 内部 Map 维护 pending resolver，`resolve()` 先删后调，防重复事件与重入问题。

---

## 2.5 `services/mcp/MCPConnectionManager.tsx`（Context 门面）

已核对要点：

1. 暴露 `useMcpReconnect()` / `useMcpToggleEnabled()`。
2. 实际连接管理由 `useManageMCPConnections` 承担。
3. 该层主要是 React context 门面，职责是把 reconnect/toggle 能力向组件树注入。

---

## 3. API 子域：重试与启动预热

## 3.1 `services/api/withRetry.ts`（重试中枢）

已核对要点：

1. `withRetry` 是统一重试生成器，返回 `AsyncGenerator<SystemAPIErrorMessage, T>`。
2. 区分前台/后台 querySource 的 529 重试策略（背景源可直接丢弃以防扩散放大）。
3. 快速模式（fast mode）下对 429/529 有单独 cooldown/fallback 路径。
4. 处理 stale keep-alive socket（`ECONNRESET/EPIPE`）时可触发 `disableKeepAlive()` 并重建 client。
5. 核心错误类型：
   - `CannotRetryError`
   - `FallbackTriggeredError`

---

## 3.2 `services/api/bootstrap.ts`（启动期 bootstrap）

已核对要点：

1. `fetchBootstrapAPI()` 在如下条件会跳过：
   - 非必要流量模式
   - 非 first-party provider
   - 无可用 OAuth/API key
2. OAuth 路径依赖 `withOAuth401Retry`，401 可自动刷新重试。
3. 响应通过 zod schema 校验，不合法数据直接丢弃。
4. `fetchBootstrapData()` 仅在缓存内容变化时持久化 global config，避免每次启动写盘。

---

## 4. Compact 子域：阈值、模式与缓存编辑

## 4.1 `services/compact/autoCompact.ts`

已核对要点：

1. `getEffectiveContextWindowSize(model)` 会预留 summary 输出 token，并允许 env 覆盖窗口。
2. `calculateTokenWarningState()` 统一给出 warning/error/autocompact/blocking 四类阈值状态。
3. `isAutoCompactEnabled()` 同时受 env 与用户配置控制。
4. `shouldAutoCompact()` 有递归/模式保护（如 `session_memory`、`compact`、context-collapse 特定路径）。
5. `autoCompactIfNeeded()` 带连续失败断路器（`MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES`），避免不可恢复场景下持续重试轰炸。

---

## 4.2 `services/compact/microCompact.ts`

已核对要点：

1. `COMPACTABLE_TOOLS` 定义可清理 tool 结果的工具集合。
2. `estimateMessageTokens()` 包含 text/tool_result/image/document/thinking/tool_use 等块的估算处理。
3. 支持 cached microcompact 路径（特性开启时）与 time-based microcompact 早触发路径。
4. `consumePendingCacheEdits/getPinnedCacheEdits/pinCacheEdits` 维护 cache-edits 生命周期，配合 API 层插入与复用。
5. 存在 main-thread source 保护，防 forked agent 污染全局 cachedMCState。

---

## 5. LSP 子域：单例状态机与重初始化

## 5.1 `services/lsp/manager.ts`

已核对要点：

1. 模块级单例 + 初始化状态机：`not-started/pending/success/failed`。
2. `initializeLspServerManager()` 是幂等入口，失败后可重试。
3. 以 generation counter 防止旧初始化 Promise 回写新状态。
4. `reinitializeLspServerManager()` 支持插件刷新后重建，避免早期缓存导致的 0 server 问题。
5. `shutdownLspServerManager()` 无论是否报错都会清理状态，防资源累积。

---

## 6. Analytics 子域：无依赖事件总线

## 6.1 `services/analytics/index.ts`

已核对要点：

1. 模块设计强调“无依赖、避免循环依赖”。
2. sink 未 attach 前，事件先进 `eventQueue`；attach 后异步 drain。
3. `stripProtoFields()` 会剔除 `_PROTO_*` 字段，避免非 1P sink 接触 PII 路由字段。
4. `logEvent/logEventAsync` 的 metadata 类型刻意限制字符串，降低误上报码/路径风险。
5. 提供 `_resetForTesting()` 清空内部状态。

## 7. 跨子域对账键（排障实用）

1. MCP：`serverName`、`session` 失效信号（404 + -32001）、工具前缀
2. API Retry：`attempt`、`querySource`、`status(429/529/401)`、cooldown 状态
3. Compact：`tokenUsage`、`threshold`、`consecutiveFailures`、cache edit 队列
4. LSP：`initializationState`、`initializationGeneration`
5. Analytics：sink attach 时机、queued_event_count、是否出现 `_PROTO_*` 外泄

---

## 8. 服务子域改动最小核对清单

1. 改 MCP config：是否影响 signature dedup 与 manual/plugin 优先级
2. 改 MCP 连接：是否破坏批量更新/重连/elicitation 注册路径
3. 改 withRetry：是否改变前后台 querySource 的 529 放大行为
4. 改 compact：是否保持断路器、主线程保护、cache-edit 生命周期一致
5. 改 LSP manager：是否保持状态机与 generation 防竞态语义
6. 改 analytics：是否保持 sink attach 前排队、attach 后 drain、PII 字段剥离

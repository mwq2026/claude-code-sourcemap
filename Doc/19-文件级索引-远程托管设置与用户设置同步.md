## 2. 远程托管设置（managed settings）

## 2.1 `services/remoteManagedSettings/index.ts`

已核对要点：

1. 核心职责：远程拉取、校验、缓存托管设置，失败时 fail-open。
2. 有 eligibility 判定入口（不同身份/订阅路径可否启用 remote managed settings）。
3. 支持 checksum + If-None-Match 缓存语义，降低重复流量。
4. 拉取含 retry/backoff，借用 API retry delay 策略。
5. 暴露“等待加载完成”能力，供其他系统在初始化阶段做依赖同步。

---

## 2.2 `services/remoteManagedSettings/securityCheck.tsx`

已核对要点：

1. 当新设置包含危险项且发生变化时，交互模式下会弹阻断确认对话框。
2. 非交互模式直接跳过对话框检查（与 trust dialog 行为一致）。
3. 用户拒绝危险变更会触发同步退出（`gracefulShutdownSync(1)`）。
4. 对话框展示/接受/拒绝均有 analytics 事件上报。

---

## 3. 用户设置同步（settings sync）

## 3.1 `services/settingsSync/index.ts`

已核对要点：

1. 交互 CLI 侧以“上传变更”为主，CCR 侧以“下载并应用”为主。
2. 上传流程：
   - 先拉远端快照
   - 构建本地 entries
   - 仅上传 changed entries（增量）
3. 下载流程：
   - 有 `downloadPromise` 缓存，首调触发，后续 join 同一 Promise
   - 提供 `redownloadUserSettings()` 强制刷新路径（不走启动缓存）
4. 同步仅在 OAuth 条件满足时启用（`user:inference` scope 足够）。
5. 失败策略为 fail-open：记录诊断/事件，不阻断主流程。

---

## 4. 同步与托管设置协同边界

1. remote managed settings 更偏“组织策略下发/受控安全入口”。
2. settings sync 更偏“用户侧设置与记忆文件跨环境对齐”。
3. 两者均采用“可用性优先，失败不阻断主链”的工程策略。

---

## 5. 改动检查清单（配置同步）

1. 改 eligibility：是否影响等待加载/初始化依赖路径
2. 改安全检查：是否仍保证危险变更在交互模式下需要显式确认
3. 改同步缓存：是否破坏首调缓存 + 强制重载语义
4. 改增量上传：是否保持 changed-only 逻辑，避免无效写放大
